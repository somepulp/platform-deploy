# ADR-001: Branch-per-Environment Strategy and Access Control

**Status:** Accepted

---

## Context

`platform-deploy` contains ArgoCD Application manifests and environment-specific Helm values that deploy application workloads to target environments. A strategy is needed to manage environment-specific configuration and control who can promote changes to each environment.

Two approaches were considered: directory-per-environment (all environments on a single branch, separated by folder path) and branch-per-environment (each environment on its own branch, ArgoCD watches different branches).

A key constraint is that `platform-deploy` is not exclusively used by the platform team — application and ML engineers also interact with it to promote changes (e.g. bumping an image tag in a lower environment). This means access control per environment is a real requirement, not just a nice-to-have.

---

## Decision

Use a branch-per-environment strategy in `platform-deploy`:

- `cluster/local` branch → local Raspberry Pi cluster
- `cluster/staging` branch → GCP staging environment (future)
- `cluster/prod` branch → GCP production environment (future)

ArgoCD watches each branch for its corresponding environment. Changes are promoted by merging a feature branch directly into the target environment branch — environment branches never merge into each other. Branch protection rules enforce access control per environment.

---

## Reasoning

### Why not directory-per-environment

Directory-per-environment was chosen for `platform-components` because platform services are managed exclusively by the platform team — no access control per environment is needed, and visibility across all environments in one place is the primary concern.

`platform-deploy` has a different audience. Application and ML engineers interact with it to promote workload changes. A developer bumping an image tag in `cluster/local` should be able to do so with minimal friction — but that same developer should not be able to promote directly to production without review. Directory structures cannot enforce this — there is no GitHub mechanism to restrict writes to a specific folder within a branch.

Branch protection rules solve this cleanly. GitHub allows per-branch protection policies that are impossible to replicate with directory structures:

- `cluster/local` — unprotected, or auto-merge for simple changes like image tag bumps
- `cluster/staging` — PR required, any reviewer approved
- `cluster/prod` — PR required, named platform team approvers only, no self-merge

### Promotion pattern

Environment branches are **independent** — they never merge into each other. Each branch maintains its own environment-specific values (image registry, ingress hostname, replica count, resource limits) that are valid only for that environment. Merging `cluster/local` into `cluster/staging` would be destructive — it would overwrite staging's registry, hostname, and configuration with local values.

Instead, changes are promoted via feature branches merged directly into the target environment branch:

```
feature/bump-ml-classifier-v1.2.0
  ↓ PR to cluster/local      ← tested on Pi, unprotected
  ↓ PR to cluster/staging    ← tested on GCP staging, reviewer required
  ↓ PR to cluster/prod       ← production, named approvers required
```

What travels between environments is the **intent** of the change (e.g. bump `ml-classifier` to `v1.2.0`), not the values file itself. Each PR to a target environment branch applies only the relevant change against that environment's existing values.

This mirrors the promotion pattern used in production GitOps platforms at scale — the same change is deliberately applied per environment with the appropriate level of scrutiny at each gate.

### Same structure on every branch

The folder structure is identical on every branch — only the values differ per environment:

```
platform-deploy/
  apps/
    ml-classifier/
      values.yaml      ← environment-specific values per branch
  README.md
```

This keeps the structure simple and predictable. ArgoCD Application manifests per environment point at the same path on different branches.

---

## Branch Protection Configuration

| Branch | Protection Rules |
|---|---|
| `cluster/local` | Unprotected — rapid iteration, auto-merge eligible for image tag changes |
| `cluster/staging` | PR required, at least 1 reviewer, any team member can approve |
| `cluster/prod` | PR required, at least 2 reviewers, named platform team approvers only, no self-merge, status checks must pass |

---

## Comparison with platform-components

| Concern | `platform-components` | `platform-deploy` |
|---|---|---|
| Primary audience | Platform team only | Platform team + app/ML engineers |
| Change frequency | Low — platform services change rarely | Higher — image tags, replica counts, config |
| Access control needed | No — platform team has full access | Yes — different gates per environment |
| Environment strategy | Directory-per-environment | Branch-per-environment |
| Reason | Visibility across envs, no access control needed | Branch protection rules enforce promotion gates |

---

## Consequences

**Positive:**
- GitHub branch protection rules enforce access control per environment — impossible with directory structures
- Promotion is a feature branch PR to each target environment — explicit, reviewable, auditable per environment
- `cluster/local` branch is unprotected — developers iterate freely without bureaucratic overhead
- Production changes require explicit approval from named platform team members
- Mirrors production GitOps promotion patterns used at scale

**Negative:**
- Branch drift risk — `cluster/local`, `cluster/staging`, and `cluster/prod` can diverge if promotions are not kept up to date
- Cross-environment visibility requires switching branches — cannot see all environments in one view
- More complex ArgoCD configuration — one Application manifest per environment pointing at different branches
